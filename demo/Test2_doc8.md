# FsWriteFile and Atomic Writes

## Purpose

`FsWriteFile` is used when the agent produces content that should be saved to disk. The most common case is writing a summary or overview document after reading and processing one or more source files. The orchestrator plans a `FsWriteFile` step after the `LlmGenerate` step that produces the summary text, using the generated text as the file contents.

## Base64 Transport

File contents travel through the FFI boundary as a base64-encoded string inside the intent JSON. This avoids any issues with null bytes, non-UTF-8 sequences, or JSON escaping of binary content. The Swift executor decodes the base64 string to `Data` and writes it to disk. For text files generated by the LLM, the content is encoded to UTF-8 bytes before base64 encoding on the Rust side.

## Intermediate Directory Creation

Before writing the file, the executor calls `FileManager.createDirectory(withIntermediateDirectories: true)` on the parent directory. This means `FsWriteFile` can create files in directories that do not yet exist without requiring a separate `FsCreateDir` step first. The directory creation is silent and idempotent.

## Overwrite Behaviour

If the target file already exists, `FsWriteFile` overwrites it. This is intentional: the agent is expected to produce a definitive output, not accumulate versions. Users who want to preserve previous outputs should rename or move them before running the workflow again. A future version may add a `FsWriteFile` option for append-only writes.

## Approval Prompt

Like all filesystem tools, `FsWriteFile` requires explicit user approval before executing. The approval prompt shows "Allow Oxcer to write: ~/path/to/file.md" so the user can see exactly what will be created or overwritten before confirming. This is particularly important for write operations since they modify the filesystem state irreversibly (within the scope of a single session).
